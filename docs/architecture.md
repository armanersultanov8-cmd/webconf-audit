# Архитектура

## 1. Назначение

`webconf-audit` - инструмент для аудита безопасности конфигураций
веб-серверов. Он выявляет ошибочные настройки, небезопасные параметры
и отклонения от распространенных рекомендаций по усилению защиты.

Инструмент поддерживает два независимых режима анализа:

- **локальный** - статический анализ конфигурационных файлов;
- **внешний** - проверка запущенного веб-узла методом черного ящика.

Оба режима входят в один продукт и поставляются в одном пакете, но
используют разные конвейеры обработки и разные источники данных.

## 2. Поддерживаемые серверы

Локальный анализ покрывает четыре веб-сервера:

- Nginx
- Apache HTTP Server
- Lighttpd
- Microsoft IIS

Внешняя проверка не зависит от конкретного сервера, но содержит
несколько сервер-специфичных проверок, которые включаются после
определения типа сервера по наблюдаемым признакам.

## 3. Принципы проектирования

1. **Разделение локального и внешнего режимов.** Это два независимых
   конвейера с разными источниками данных и разной семантикой. Общей
   для них остается только модель результата и отчета.

2. **Статический локальный анализ.** Локальный режим не запускает
   веб-сервер и не зависит от работающего окружения.

3. **Минимальная нормализация.** Единая модель данных применяется
   только для сущностей, которые можно напрямую сравнивать и которые
   используются универсальными правилами. В остальных случаях каждый
   сервер сохраняет собственное представление.

4. **Трассируемость.** Каждая находка и каждая техническая проблема
   анализа связана со своим источником:
   - путь к файлу и номер строки для текстовых конфигураций;
   - путь к файлу и XML-узел или секция для IIS;
   - наблюдаемая конечная точка, URL, заголовок или TLS-параметр для
     внешнего анализа.

5. **Изоляция серверной семантики.** Глубокая логика конкретных
   серверов находится в сервер-специфичных парсерах, вспомогательных
   функциях и наборах
   правил. Общий конвейер не должен знать, как каждый сервер объединяет
   свою конфигурацию.

6. **Находки отделены от технических проблем.** Проблема безопасности
   и техническая проблема анализа - разные сущности. Они разделены на
   уровне модели и отчета.

## 4. Конвейер локального анализа

```text
входная конфигурация -> обнаружение источников -> парсинг
   -> include/наследование -> эффективная конфигурация
   -> нормализация -> выполнение правил -> результат
```

1. **Входная конфигурация.** Пользователь передает основной
   конфигурационный файл или корневой источник конфигурации.

2. **Обнаружение источников.** Анализатор строит контекст загрузки:
   полный набор прочитанных файлов и связи include между ними.

3. **Парсинг.** Сервер-специфичный парсер строит AST или эквивалентную
   структурированную модель.

4. **Разрешение include и наследования.** Для текстовых серверов
   раскрываются директивы `include`; для IIS восстанавливается цепочка
   наследования между конфигурационными файлами.

5. **Эффективная конфигурация.** Строится там, где плоского AST
   недостаточно:
   - Apache - `.htaccess`, `AllowOverride`, `VirtualHost`, `Location`,
     семантика объединения заголовков;
   - Lighttpd - глобальные директивы и условные области;
   - IIS - `machine.config` -> `applicationHost.config` -> `web.config`,
     а также наследование `<location>`.

6. **Минимальная нормализация.** Сервер-специфичные представления
   преобразуются в небольшую общую модель, содержащую только сущности,
   нужные универсальным правилам.

7. **Выполнение правил.** Последовательно работают два слоя:
   - сервер-специфичные правила по AST или эффективной конфигурации;
   - универсальные правила по нормализованной модели.

8. **Отчетность.** Анализатор возвращает `AnalysisResult`, который
   затем передается в слой формирования отчета.

## 5. Конвейер внешнего анализа

```text
цель -> поиск портов -> HTTP/HTTPS-проверки -> анализ TLS-параметров
   -> определение сервера -> проверка чувствительных путей
   -> выполнение правил -> результат
```

1. Разбор цели и, при необходимости, поиск портов.
2. HTTP- и HTTPS-проверки с резервным переходом `HEAD` -> `GET` и
   отдельным потоком `OPTIONS`.
3. Анализ TLS-параметров: согласованный протокол и шифр, активно проверенные
   поддерживаемые версии TLS, полнота цепочки сертификатов, извлечение
   SAN.
4. Определение сервера по заголовкам ответа, стандартным страницам
   ошибок и реакции на намеренно некорректные запросы.
5. Проверка чувствительных путей.
6. Выполнение внешних правил по собранным наблюдениям.
7. Формирование отчета через те же модели результата и отчета, которые
   используются локальным режимом.

Внешний режим переиспользует модели результата и отчета, но не зависит
от AST и модулей эффективной конфигурации локального режима.

## 6. Сервер-специфичные заметки

### 6.1 Nginx

- Текстовый парсер с раскрытием `include`.
- Анализатор, работающий в первую очередь с AST.
- Поддержка glob-шаблонов, обнаружение циклов и отслеживание исходной
  позиции каждой директивы.

### 6.2 Apache

Конфигурация Apache состоит из двух слоев:

- основной конфигурационный файл и его `Include` / `IncludeOptional`
  файлы;
- распределенные переопределения через `.htaccess`.

Анализатор Apache обрабатывает оба слоя:

- поиск `.htaccess` через блоки `Directory` и `DocumentRoot`;
- фильтрация директив `.htaccess` по `AllowOverride`;
- вспомогательные функции эффективной конфигурации для `VirtualHost`,
  `Location` и `LocationMatch`;
- применимость `ServerName` и `ServerAlias`;
- семантика объединения накопительных директив `Header`;
- отдельные контексты анализа для каждого `VirtualHost` и
  нормализация по контекстам.

### 6.3 Lighttpd

- Парсер строит структурированную модель, сохраняющую условные блоки.
- Эффективная конфигурация хранит глобальные директивы и условные
  области отдельно.
- Таргетированный анализ может учитывать контекст, похожий на запрос,
  например конкретный `host`.
- `include_shell` безопасен по умолчанию: пропускается с
  предупреждением; явное выполнение включается отдельно.

### 6.4 Microsoft IIS

- XML-парсинг через `defusedxml` создает документ и структурированные
  секции.
- Эффективная конфигурация восстанавливает цепочку наследования
  `machine.config` -> `applicationHost.config` -> `web.config`, а
  также семантику коллекций `<add>` / `<remove>` / `<clear>`.
- Метаданные результата фиксируют цепочку происхождения и цепочку
  наследования.
- Настройки TLS-протоколов и шифров часто находятся вне XML-
  конфигурации, например в реестре Windows, поэтому локальный
  анализатор не восстанавливает их полностью.

## 7. Нормализация

Нормализация не пытается свести все четыре сервера к одной общей
абстракции. Она извлекает только те сущности, которые напрямую
сопоставимы между серверами:

- точки прослушивания;
- TLS-намерение и TLS-настройки;
- заголовки безопасности;
- маркеры политики доступа;
- раскрытие идентификации сервера.

Нормализаторы для отдельных серверов находятся в
`src/webconf_audit/local/normalizers/`. Универсальные правила используют
полученный `NormalizedConfig`. Неизвестные значения остаются `None`, а
соответствующие универсальные правила молчат: цель - избежать ложных
срабатываний из-за отсутствующих данных.

## 8. Система правил

Система правил построена вокруг централизованного реестра.

Ключевые элементы:

- `RuleMeta` - метаданные правила;
- `RuleEntry` - `RuleMeta` и вызываемый объект, реализующий правило;
- `RuleRegistry` - каталог всех известных правил и хранилище
  исполняемых правил;
- `@rule` - декоратор для регистрации модульных правил;
- `ensure_loaded(...)` - ленивый загрузчик пакетов, импортирующий
  модули правил только при необходимости;
- `register_meta(...)` - регистрация метаданных для записей каталога,
  у которых нет исполняемой реализации; используется для внешних
  правил, представленных только метаданными.

Текущий каталог содержит 183 правила.

| Локальные - Nginx | 41 |
| Локальные - Apache | 27 |
| Локальные - Lighttpd | 15 |
| Локальные - IIS | 20 |
| Универсальные (локальные) | 11 |

## 9. Модель отчетности

Локальный и внешний режимы используют одну модель результата:

- `Finding` - проблема безопасности.
- `AnalysisIssue` - техническое предупреждение или проблема,
  возникшая во время анализа.
- `AnalysisResult` - результат одного запуска анализатора.

Общий слой отчетности добавляет:

- `ReportData` - агрегированные результаты одного или нескольких
  запусков анализатора.
- `ReportSummary` - итоги по серьезности, режиму и типу сервера.
- `TextFormatter` - человекочитаемый вывод.
- `JsonFormatter` - машинно-читаемый вывод.

Возможности отчетности:

- сортировка находок по серьезности;
- сортировка технических проблем по уровню;
- `--format text|json` для каждой команды `analyze-*`;
- находки универсальных правил подавляются, если более конкретное
  серверное правило уже сообщило о той же проблеме в том же месте.

## 10. Известные ограничения

- Анализ Lighttpd намеренно консервативен: моделируются не все
  документированные операторы условий и цепочки `else if`.
- Локальный анализ IIS не восстанавливает настройки TLS-протоколов и
  шифров, заданные вне XML-конфигурации, например в реестре Windows /
  SChannel.
- Токенизатор Nginx пока не поддерживает аргументы директив в одинарных
  кавычках.
